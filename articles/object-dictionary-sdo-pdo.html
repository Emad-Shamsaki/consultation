Demystifying the Object Dictionary, SDO, and PDO in Industrial Automation

A practical explanation of what each object does and how devices exchange configuration values and process data in a CANopen network.

Emad Shamsaki
Sep 2, 2025 | CANopen | 8 min read

Many CANopen systems feel difficult at first because three ideas are introduced at the same time: the object dictionary, SDO transfers, and PDO mapping. Once these are separated, the behavior becomes much easier to reason about.

Object dictionary

The object dictionary is the structured memory map of the device. Every configuration value, measured variable, and status field is given an index and subindex. That means communication is not built around raw addresses but around well-defined data objects.

SDO vs PDO

Use SDO when you need to read or write configuration data. Use PDO when you need regular real-time exchange of operational data such as setpoints, states, or measurements.

A good mental model is simple: SDO is for configuration and inspection, PDO is for live traffic.

Typical integration sequence

  1. Read the EDS or device manual and identify the required objects.
  2. Configure parameters with SDO.
  3. Map the required process signals into PDOs.
  4. Verify timing and state transitions during startup.
